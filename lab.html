Freeze the task before touching the harness

The lab's primary example is TD-A-01-RETRY-001: a coding agent must fix a retry-state bug in a small module and pass a named acceptance test. The task, repository revision, permitted tools, test commands, retry budget, and acceptance rule are recorded before anything else changes. The reproducible bad baseline and passing fix live in challenge/.

A lower token count with a failed patch is not an improvement. The acceptance test is the denominator for every claim that follows.

This mirrors the rule production teams use: define the verified code change and quality gate first, then optimize the path underneath it. The agent-optimizer skill is that contract written down.

What the harness actually loaded

The user request is one sentence. The prompt the model processes is not: system instructions, tool schemas, repository context, prior turns, and tool logs. On the frozen task the baseline harness context is 40,176 characters, of which 17,420 are repeated across 14 tool calls — 43% of the bill is replay.

Count tokens the way the provider bills them

A tokenizer maps text into integer token IDs. A token is a model unit, not a word. Cached input, cache-creation input, and fresh input are different line items, and the two big providers account for them differently:

Accounting · from the agent-optimizer skill
Claude Code:  total input = input_tokens
                          + cache_creation_input_tokens
                          + cache_read_input_tokens

Codex:        total input = input_tokens
              uncached    = input_tokens - cached_input_tokens
              # cached_input_tokens is a subset — never add it again

A cache read is cheaper, not free. Mixing the two ledgers — or adding a subset back to its total — manufactures savings that do not exist.

Context lifetime is the control surface

Classify every span of context as stable (system, schemas, repo prefix), active (the current diff and failing test), durable (decisions worth persisting), externalized (retrievable on demand), or dead (logs replayed out of habit). Then change one policy at a time — stable-prefix layout, tool-output projection, skill disclosure, repository memory — and re-run the same task.

Local MLX-LM or vLLM on Nebius

The lab's engine decision flips live. For one private user on a MacBook, MLX-LM wins: no credentials leave the machine, unified memory holds the model, setup cost is minutes. Switch the slider to one hundred concurrent users and the recommendation changes to vLLM on Nebius: continuous batching, paged KV, and provisioned GPUs beat a laptop queue.

Rank privacy, setup cost, hardware fit, concurrency, and acceptance quality before runtime features. A production serving engine is not always faster than a local runtime — that misconception is literally the EverOS memory fixture this lab replays in phase twelve.

Claim boundary. The Nebius branch is fixture_backed until a Token Factory key or pinned vLLM endpoint captures a real provider request ID.

Layout decides the hit rate

System instructions and repository context are byte-identical across turns — if the harness keeps them at the front, in the same order. A prompt-cache hit can avoid repeated prefix computation; moving one retrieved document reorders the prefix and turns the same bytes into a miss. This is why "stable-prefix layout" is the first policy the lab applies.

The take-home path runs this loop on your own Mac and mimics the observability columns production systems expose — lookup_requested_tokens vs lookup_hit_tokens — in the mirrored post Set Up a Local OpenClaw or Hermes KV-Cache Learning Loop with Mac MLX.

Every input token is paid at prefill

Prefill processes the supplied input tokens and creates attention key/value state. It parallelizes well and is compute-intensive. The waste pattern in agent loops is repeated prefill over the same repository prefix: the tool loop appends a result, the harness rebuilds the prompt, and the system pays for 30,000 characters of unchanged prefix again.

The lab's causal path
user task -> harness context -> tokens -> prefill -> KV cache
          -> decode -> HBM residency -> acceptance test

Reuse beats recompute — when it arrives in time

Each transformer layer stores key and value vectors for prior tokens. Keeping them means a new token attends to old context without re-running the full prefix. A 50-turn agent is mostly old state plus a small new delta; if the stack cannot find the old state cheaply, it recomputes it.

The break-even rule the lab teaches: a cache or offload path wins when lookup + transfer + restore + miss risk < recomputed prefill under the required latency. That rule is the same on a 16GB MacBook and a GB200 rack — only the tiers change. Deep dive: KV Cache Is Becoming the Memory Hierarchy of Inference.

Small steps, repeated reads

Decode produces tokens incrementally, re-reading weights and the growing KV state at every step. It is usually memory-bandwidth-bound, not compute-bound — which is why output tokens cost several times more than input tokens on every provider price sheet, and why verbose agent output is a direct tax on the loop.

The optimization is not "generate faster" — it is generate less that the verifier throws away: tighter diffs, structured tool calls, no restated context in the reply.

Where the bytes physically sit

On the production branch, model weights, active KV, and runtime workspaces occupy GPU HBM. On the local branch, Apple unified memory holds the same roles in one pool shared with macOS and the browser. Either way the lesson holds: every replayed character eventually occupies the most expensive memory in the system, and occupancy you waste is concurrency you cannot serve.

Claim boundary. The lab shows placement functionally. It does not read HBM counters; local Mac columns are proxies (observability_mimic: true), never live LMCache, vLLM, or SGLang telemetry.

Verification is the denominator

The lab re-runs the same task against the same revision and the same acceptance contract after each policy change. 10/10 local tests pass; the verifier script checks required artifacts, JSON contracts, and a secret-safe bundle on every run. A patch that fails tests makes every token spent on it waste — retroactively.

Cost per accepted task, not cost per token. Retries and rejected work stay in the numerator; only verified successes enter the denominator.

Retrieve the Case, not the transcript

After a verified run, the trajectory is compressed into an EverOS Case. The next session retrieves 1,290 characters (~323 estimated tokens) instead of replaying the full 16,840-character (~4,210 token) trajectory — a −92% reduction, computed locally as characters ÷ 4. The recalled misconception ("a production engine must always be faster than a local runtime") visibly changes the next lesson's instruction.

Claim boundary. Local estimate on fixture data. EverMind's own 7–15× / up-to-90% figures are vendor claims and are never presented as Touchdown measurements.
